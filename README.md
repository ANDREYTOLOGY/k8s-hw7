# Домашнее задание занятию `«Расчёт ресурсов Kubernetes-кластера»` - `Чернышов Андрей`

## Исходные данные

| Компонент   | Количество реплик | RAM на экземпляр | CPU на экземпляр |
| ----------- | ----------------- | ---------------- | ---------------- |
| База данных | 3                 | 4 ГБ             | 1 vCPU           |
| Кэш         | 3                 | 4 ГБ             | 1 vCPU           |
| Backend     | 10                | 600 МБ           | 1 vCPU           |
| Frontend    | 5                 | 50 МБ            | 0.2 vCPU         |

---

## Расчёт требуемых ресурсов

### База данных

* RAM: 3 × 4 ГБ = 12 ГБ
* CPU: 3 × 1 = 3 vCPU

### Кэш

* RAM: 3 × 4 ГБ = 12 ГБ
* CPU: 3 × 1 = 3 vCPU

### Backend

* RAM: 10 × 600 МБ = 6000 МБ ≈ 6 ГБ
* CPU: 10 × 1 = 10 vCPU

### Frontend

* RAM: 5 × 50 МБ = 250 МБ ≈ 0.25 ГБ
* CPU: 5 × 0.2 = 1 vCPU

### Итого для приложения

| Ресурс | Значение |
| ------ | -------- |
| RAM    | 30.25 ГБ |
| CPU    | 17 vCPU  |

---

## Служебные ресурсы Kubernetes

Для работы компонентов Kubernetes (CoreDNS, kubelet, ingress-controller, metrics-server и др.) добавим резерв:

* RAM: 2 ГБ на каждую ноду
* CPU: 1 vCPU на каждую ноду

---

## Выбор количества рабочих нод

Для обеспечения отказоустойчивости база данных и кэш должны быть распределены минимум по трём узлам.

Выберем 3 worker-ноды.

Параметры одной ноды:

* 8 vCPU
* 16 ГБ RAM

Суммарные ресурсы кластера:

| Ресурс | Значение |
| ------ | -------- |
| CPU    | 24 vCPU  |
| RAM    | 48 ГБ    |

После вычета служебных ресурсов:

| Ресурс | Доступно приложениям |
| ------ | -------------------- |
| CPU    | 21 vCPU              |
| RAM    | 42 ГБ                |

Это превышает расчётную нагрузку:

* Требуется: 17 vCPU и 30.25 ГБ RAM
* Доступно: 21 vCPU и 42 ГБ RAM

---

## Проверка отказоустойчивости

При выходе из строя одной ноды остаётся:

* 2 ноды × 8 vCPU = 16 vCPU
* 2 ноды × 16 ГБ = 32 ГБ RAM

После вычета служебных ресурсов:

* CPU: 14 vCPU
* RAM: 28 ГБ

Такого объёма уже недостаточно для полной нагрузки приложения (не хватает CPU и RAM).

Поэтому для соблюдения требования отказоустойчивости необходимо добавить запас.

---

## Итоговый вариант

Используем 4 worker-ноды со следующими характеристиками:

* 8 vCPU
* 16 ГБ RAM

Общие ресурсы:

* CPU: 32 vCPU
* RAM: 64 ГБ

После вычета служебных ресурсов:

* CPU: 28 vCPU
* RAM: 56 ГБ

При отказе одной ноды остаётся:

* CPU: 21 vCPU
* RAM: 42 ГБ

Что превышает требуемые ресурсы приложения:

* 17 vCPU
* 30.25 ГБ RAM

---

## Итог

Для размещения приложения рекомендуется использовать кластер из:

* 1 master-node
* 4 worker-node

Параметры каждой worker-ноды:

* 8 vCPU
* 16 ГБ RAM

Такой кластер обеспечивает:

* размещение всех компонентов приложения;
* распределение отказоустойчивых сервисов между узлами;
* сохранение работоспособности при отказе одной worker-ноды;
* наличие резерва ресурсов для обновлений и масштабирования.
